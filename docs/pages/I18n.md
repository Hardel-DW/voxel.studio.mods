# I18n — Translation System
All dynamic text resolution goes through `StudioText`. Static UI labels use `I18n.get()` directly.

### StudioText API
Two public overloads, one resolution chain:

```java
StudioText.resolve(String domain, Identifier id)
StudioText.resolve(ResourceKey<? extends Registry<?>>, Identifier id)
```

Resolution chain (same for both):
1. **Item special case** — `BuiltInRegistries.ITEM.getValue(id).getName()` (only when domain = "item")
2. **Custom key** — `I18n.get("domain:namespace:path")` (our lang JSON files)
3. **MC key** — `I18n.get("domain.namespace.path")`
4. **Humanize** — `fire_aspect` → `Fire Aspect` (private fallback, never callable directly)

### Translation Key Conventions
Identifiers use the standard Minecraft format: `namespace:path`.

| Type | Format | Example |
|------|--------|---------|
| Registry entry | `{registry}:{identifier}` | `enchantment:minecraft:sharpness` |
| Tag | `{registry}_tag:{identifier}` | `item_tag:minecraft:enchantable/sword` |
| Effect | `effect:{identifier}` | `effect:minecraft:damage` |
| Slot (fixed set) | `slot:{id}` | `slot:mainhand` |
| Description suffix | `+ .desc` | `effect:minecraft:damage.desc` |
| Static UI | free form | `foo:bar.baz` |

### Rules
- `components/ui` must never call `I18n` — it stays generic and receives pre-translated strings. Translation is the page's responsibility.
